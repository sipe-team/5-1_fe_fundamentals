## PR 작성 가이드

PR 본문에 아래 항목들을 포함해주세요. 코드만큼이나 **설계 판단의 근거**가 중요합니다.

### 1. 시간 충돌 검증 전략
> 시간 충돌을 어떻게 검증했는지 설명해주세요.

시작 시간과 종료 시간 간의 기본적인 충돌은 zod.refine을 활용한 커스텀 유효성 검증으로 처리했습니다.
이를 통해 폼 제출 이전 단계에서 잘못된 시간 입력(시작 ≥ 종료, 과거 시간 등)을 사전에 차단했습니다.

다른 예약과의 시간 충돌은 서버에서 최종 검증을 수행하고 있어, 프론트에서는 해당 요청의 응답 코드(409 Conflict)를 기준으로 에러를 처리했습니다.
이때 서버 응답을 기반으로 사용자에게 충돌 안내 메시지를 노출하는 방식으로 역할을 분리했습니다.

→ 입력 값 검증은 프론트, 데이터 정합성은 서버에서 책임지도록 설계했습니다.

### 2. 상태 관리 전략
> 예약 데이터를 어떻게 관리하는지? (TanStack Query, useState, 전역 store 등)
> 예약 생성/취소 후 여러 페이지의 데이터를 어떻게 동기화하는지?

서버 상태는 TanStack Query를 사용해 관리했습니다.
예약 목록, 회의실 목록 등은 모두 query로 관리하고, UI 상태(필터, 에러 등)는 useState로 분리했습니다.

예약 생성/삭제 이후에는 invalidateQueries를 통해 관련 데이터를 무효화하고,
재요청을 통해 최신 상태를 반영하도록 했습니다.

→ 서버 상태를 직접 수정하지 않고, query 기반의 재동기화 전략을 선택했습니다.

### 3. 폼 상태 관리
> 예약 생성 폼의 상태를 어떻게 관리했는지?
> 유효성 검증은 어디서, 어떻게 처리했는지?

폼 상태 관리는 react-hook-form + zod 조합을 사용했습니다. 유효성 검증은 다음과 같이 구성했습니다.
- 기본 입력 검증: zod schema
- 복합 조건 검증: zod.refine
- 일부 UI 제약: input[type="time"]의 min, max, step

다만 시간 제약과 관련된 검증이 네이티브와 zod로 분산된 점은 아쉬운 부분입니다.
PR 이후에는 입력 방식 자체를 개선하고, zod 기반 검증으로 일원화할 예정입니다

### 4. 컴포넌트 구조
> 어떤 기준으로 컴포넌트를 나눴나요?
> 폴더 구조는 어떻게 잡았고, 왜 그렇게 했나요?

현재는 서비스 규모가 크지 않아 기존 구조를 유지했습니다.
다만 components 하위에 폴더를 세분화했지만, 각 폴더에 단일 컴포넌트만 존재하는 경우가 많아 과도하게 분리한 것 같습니다.

### 5. 과제를 수행하는데 어려움은 없었나요?
> 어려웠던 부분이 있다면 말씀해주세요~! 추후 과제때 개선해보겠습니다.

타임라인 UI(TimeLineGrid)를 구현하는 과정이 가장 어려웠습니다.
레이아웃 계산과 시간 단위 배치 로직이 복잡해 일부는 AI의 도움을 받아 구현했습니다.

AI가 구현해 준건 과도한 계산식이 많아 결국 폐기했습니다.
다행히 flex 서비스에서 유사한 타임라인을 사용하고 있어 스타일링 작업에 큰 도움을 받았습니다


### 6. 추가 라이브러리 (optional)
> 추가로 설치한 라이브러리가 있다면, 각각 왜 선택했는지 설명해주세요.

`react-router-dom`: 라우팅 처리, URL 파라미터 관리를 쉽게 진행하기 위해
`tanstack/react-query`: 서버 상태 관리와 비동기 데이터 흐름을 일관되게 처리하기 위해 선택했습니다.
특히 캐싱, invalidation, refetch 등 데이터 동기화 문제를 단순화할 수 있다는 점이 장점이었습니다.
`tailwinds`: 빠른 UI 구현과 일관된 스타일 관리를 위해 사용했습니다.
`react-hook-form`: 간결하게 폼 상태를 관리하기 위해 선택했습니다.
`zod` :타입 정의와 유효성 검증을 하나의 스키마로 통합하기 위해 사용했습니다. 특히 react-hook-form과 궁합이 좋아 선택했습니다.

### 7. 아쉬운 점 / 개선하고 싶은 점 (optional)
> 시간이 더 있었다면 어떤 부분을 개선했을지 자유롭게 적어주세요.

1. 신규 예약 폼이 가장 아쉽습니다.
신규 예약 폼의 UX가 가장 아쉬운 부분입니다. 특히 input[type="time"]의 step 속성이 Mac에서 제대로 동작하지 않는 문제가 있었습니다.
select 기반으로 30분 단위 옵션을 제공하는 컴포넌트를 제작하는게 더 사용성에 좋을 것같습니다.
유효성 검사가 native와 zod refine 두 곳으로 분산되었는데 input도 손보고 유효성 검증을 zod로 통일적으로 처리할 수 있도록 리팩토링을 진행하고 싶습니다.

2. 타임라인에서 날짜를 변경했을 때, 전체를 다시 렌더링하는 구조도 개선이 필요합니다.
회의실(rooms)은 변경이 적은 데이터이므로 유지하고, 예약 데이터만 갱신하는 방식으로 최적화하면 좋을 것 같습니다.
